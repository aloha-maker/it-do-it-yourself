## 🚀 GitHub 開発フロー標準手順書
プロジェクトの開発効率を高め、コードの品質を維持するための基本ワークフローです。

### 1. タスクの起票 (Issue)　担当：PM
まずは作業内容を明確にします。

* **GitHub Issues** を開き、「New Issue」を作成します。
* **テンプレートの活用**: バグ報告や機能要望など、適切なテンプレートを選択します。
* **詳細の記載**: 現状の課題、期待される動作、影響範囲を記述します。
* **Assignees / Labels**: 担当者（自分）を割り当て、適切なラベル（`bug`, `feature`など）を付与します。

### 2. ローカル環境の準備　担当：作業者

最新の状態から作業を開始します。

```bash
# mainブランチへ移動
git checkout main

# 最新の変更をプル
git pull origin main

```

### 3. ブランチの作成　担当：作業者

タスクごとに専用の作業ブランチを作成します。

```bash
# 命名規則例: feature/issue-番号_概要
git checkout -b feature/issue-123_user-auth

```

### 4. 実装とコミット　担当：作業者

作業を進め、意味のある単位でコミットします。

* **Atomic Commit**: 1つのコミットには1つの変更だけを含めるように意識します。
* **コミットメッセージ**: `feat: ログイン機能の実装 (#123)` のように、Issue番号を含めると紐付けがスムーズです。

```bash
git add .
git commit -m "feat: ログインバリデーションの追加 (#123)"

```

### 5. リモートへのプッシュ　担当：作業者

ローカルの変更をGitHub上にアップロードします。

```bash
git push origin feature/issue-123_user-auth

```

### 6. プルリクエスト (PR) の作成　担当：作業者

コードレビューを依頼するためのPRを作成します。

* **Base & Compare**: `base: main` ← `compare: [作業ブランチ]` になっているか確認します。
* **Description**:
* **何を・なぜ変更したか**: 簡潔に記述します。
* **Issueとの紐付け**: 説明欄に `Closes #123` と記述すると、PRマージ時にIssueが自動で閉じられます。


* **Reviewers**: チームメンバーをレビュー担当者として指名します。

### 7. レビューと修正　担当：PM & 作業者

レビュアーからのフィードバックに対応します。

* **指摘事項の修正**: 指摘があればローカルで修正し、再度 `push` します（PRに自動反映されます）。
* **Approveの獲得**: 全ての指摘が解消され、Approve（承認）を得るまで継続します。

### 8. マージと後片付け　担当：PM

承認されたら、いよいよ統合です。

* **Merge**: GitHub上の「Merge pull request」ボタンを押します（プロジェクト方針により `Squash and merge` を推奨）。
* **ブランチ削除**: 不要になったリモートブランチを削除します。
* **ローカルの更新**:
```bash
git checkout main
git pull origin main
# ローカルの作業ブランチを削除
git branch -d feature/issue-123_user-auth

```
